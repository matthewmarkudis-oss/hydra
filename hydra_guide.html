1. What is Hydra?

Hydra is a multi-agent reinforcement learning (MARL) system where multiple AI agents collaborate to manage a simulated portfolio. Instead of hand-coded rules, agents learn strategies through trial and error in a simulated market environment.

The core idea comes from two research directions:

  • Google DeepMind — Training agents against a diverse pool (learning + static + rule-based) produces cooperative behavior without hardcoded coordination
  • FinRL / Covalent — Gym-compatible environments with RL algorithms (PPO, SAC, A2C) and heterogeneous CPU/GPU compute orchestration

The system operates on simulated intraday data (5-minute bars) with synthetic market environments for training. Each episode represents one simulated day (~78 bars from 9:30–16:00).

2. The Agent Pool

Hydra uses a diverse population of 6 agents. Diversity is key — different agent types bring different strengths, and cooperation emerges naturally from shared rewards.

PPO Agent
Learning
Proximal Policy Optimization. The workhorse of RL. Clips policy updates to prevent destructive large steps. Good at learning stable strategies. Uses neural network policy (MlpPolicy) with SB3.
SAC Agent
Learning
Soft Actor-Critic. Maximizes both reward and entropy (exploration). Better at finding diverse strategies and avoiding local optima. Off-policy = more sample efficient.
A2C Agent
Learning
Advantage Actor-Critic. Simpler on-policy method. Fast to train, lower variance than vanilla policy gradient. Good complement to PPO and SAC.
Static Agent
Static
Frozen snapshot. A previously-trained RL agent that no longer updates its weights. Provides a stable reference strategy that learning agents train against. Cannot be fooled into changing.
Alpha Momentum
Rule-Based
Momentum strategy adapter. Wraps the existing Alpha agent from the old system. Follows trends — buys when price is rising, sells when falling. Provides a non-RL baseline.
Beta Mean Reversion
Rule-Based
Mean reversion adapter. Wraps the existing Beta agent. Bets that prices revert to average — buys dips, sells rallies. Counterbalances momentum strategies.

Why Mix Agent Types?

The DeepMind insight: training RL agents against only other RL agents leads to unstable co-evolution. Including static agents (frozen weights) and rule-based agents (handcrafted logic) anchors the learning and prevents strategy collapse. Learning agents must develop strategies that work alongside agents they can't influence.

3. How It Works

Shared Portfolio

All 6 agents observe the same market state and independently choose actions. Their actions are aggregated via weighted average (equal weight, normalized), producing a single portfolio action. The shared reward (based on the combined portfolio's performance) incentivizes cooperation — agents learn to complement each other.

The Observation (What Agents See)

~75-dimensional float32 vector per timestep:

CategoryFeatures
PortfolioCash position (normalized), per-stock holdings
Price DataPer-stock current price (normalized)
Technical IndicatorsRSI, MACD, CCI, Bollinger %B, volume ratio (per stock)
Global ContextVIX level, session type (open/mid/close), time-of-day, current drawdown, daily P&L

The Action (What Agents Do)

Continuous value in [-1, +1] per stock:

  • Negative = Sell that fraction of current holdings
  • Positive = Buy that fraction of available cash
  • Zero = Hold

The Reward (How Agents Learn)

Reward = Differential Sharpe Ratio + penalties

  • Differential Sharpe: Measures risk-adjusted return improvement at each step
  • Drawdown penalty: Punishes large portfolio declines
  • Transaction cost penalty: Discourages excessive buying/selling

Risk constraints (position limits, daily loss cap, drawdown halt) are enforced inside the environment — agents learn to respect them because violations produce immediate negative consequences.

Population-Based Training (Generations)

Training is organized into generations. Each generation:

  1. All agents play through episodes together (multi-agent environment)
  2. Track per-agent and portfolio metrics
  3. Learning agents do SB3 gradient updates (agent.train_on_env())
  4. Evaluate all agents
  5. Promote top performers / freeze as static / demote worst

This is inspired by DeepMind's population-based training — the pool evolves over time, and agents must adapt to an ever-changing mix of opponents and teammates.

Each seed generates a different synthetic market with unique price dynamics, volatility patterns, and trend characteristics. An agent that only performs well on one seed is overfitting. We want agents that consistently pass validation across many seeds.

6. How Do We Know If It's Working?

This is the most important section. Here are the concrete metrics that tell you whether the system is succeeding:

Key Success Metrics

Sharpe Ratio > 0.5

Risk-adjusted return. Measures return per unit of risk. Above 0.5 means the strategy is generating meaningful alpha relative to its volatility. Above 1.0 is strong.

ATHENA Validation Pass

Walk-forward out-of-sample test. Agents must pass on data they never trained on. This is the gold standard — it prevents overfitting to training data.

Seed Consistency > 70%

An agent type passing validation on 7+ of 10 seeds means it generalizes well. If it only passes on 2-3 seeds, it's likely overfitting to specific market conditions.

Reward Improvement Across Generations

Episode rewards should trend upward (less negative, then positive) as training progresses. Flat or declining rewards mean the agent isn't learning.

What "Good" Looks Like

MetricBadOkayGoodGreat
Mean Reward < -100 -100 to -30 -30 to 0 > 0
Sharpe Ratio < 0 0 to 0.5 0.5 to 1.0 > 1.0
Max Drawdown > 20% 10-20% 5-10% < 5%
Seeds Passing 0-2 / 10 3-6 / 10 7-8 / 10 9-10 / 10
ATHENA Validation 0 agents 1-2 agents 3-4 agents 5-6 agents

Warning Signs

Flat Rewards

If rewards don't improve across generations, the learning rate may be wrong, the reward signal too sparse, or the environment too hard.

High Variance Across Seeds

If an agent passes on seed 42 but fails on seed 43, it's overfitting. Robust strategies should work across diverse market conditions.

All Agents Agree

If all agents take identical actions, they've converged to a single strategy. The pool has lost diversity. Static and rule-based agents should prevent this.

Excessive Holding

If agents learn to just hold cash (0 activity), they've found a local optimum that avoids losses but doesn't generate returns. Check the transaction count metric.

Quick Health Check

  1. Open http://localhost:6006
  2. Check Learning/Episode_Reward — is it trending up?
  3. Check Risk/Sharpe_Ratio — is it above 0?
  4. Check Trading/Trades_per_Episode — is the agent active?
  5. If all three are positive, the system is learning.

RAMSystem memoryMarket data caching, replay buffers

GPU Note

The AMD 6900XT uses DirectML (not CUDA). SB3 (stable-baselines3) is incompatible with DirectML because it internally converts tensors to numpy, which requires CPU memory. So SB3 training runs on CPU. The GPU is available for custom PyTorch operations outside of SB3 via torch_directml.device(). Future work: custom training loops that bypass SB3 to leverage the GPU fully.

9. How to Run

Environment Setup

conda activate hydra          # Python 3.11 environment
cd C:\TradingAgents\hydra

Single Training Run

python scripts/train.py

Runs one full pipeline (data prep -> train -> evaluate -> validate) with default settings.

Full Curriculum (10 Seeds)

python scripts/train_curriculum.py

Runs all 3 phases (warmup, main, stress) across 10 seeds. Results saved to logs/curriculum_results.json.

Custom Seeds

python scripts/train_curriculum.py --seeds 42 100 200 --generations 5 --episodes 20

Launch Dashboard

tensorboard --logdir=logs/tensorboard --port=6006

Then open http://localhost:6006 in your browser.

Run Tests

python -m pytest tests/ -v

Key Design Principle

Risk constraints are enforced inside the environment, not as post-processing. The agent experiences the consequences of violating limits (clipped actions, terminal states, zero reward) during training, so it learns to respect them naturally. This is fundamentally different from rule-based systems that apply limits after decisions are made.
